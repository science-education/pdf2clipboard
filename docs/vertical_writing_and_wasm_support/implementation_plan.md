# 縦書き表セル文字の配置改善およびWebAssembly化 実装計画

## 概要
本プロジェクトにおいて、PDF内の表セル等で縦書き文字（WMode=1）がマス目の中央にならず右上に寄ってしまう問題の修正、および不足文字のフォールバック強化を行う。さらに、アプリケーション全体をWebAssembly (Wasm) としてブラウザ上で動作可能にするためのアーキテクチャ対応を実施する。

## 主な課題と解決策

### 1. 縦書きモード（WMode=1）における文字配置と文字送りの適正化
- **問題**: 標準的なPDF展開において、縦書き時には文字の基準原点が左下から上端中央にシフトするが、現行の描画ロジックでは原点補正が行われておらず、文字送りの軸も横方向のままであった。
- **解決策**:
  - `render_unicode_text` および `render_cid_direct` において、WMode=1 の場合に標準的なCJK縦書き原点オフセット（左へ約0.5em、下へ約0.88em）を適用。
  - 描画後のカーソル移動（文字送り）を横軸（`x_cursor`）から縦軸（`y_cursor`）への減算へと切り替え、縦方向へのレイアウトを適正化する。

### 2. 未定義グリフに対する段階的フォールバックの完全統合
- **問題**: 埋め込みフォントのCIDマッピングや直接描画でグリフが存在しない場合、文字が表示されずに消失していた。
- **解決策**:
  - `render_cid_direct` および `render_unicode_text` の双方において、対象文字のUnicode値に基づき、Stage 1 (Latinフォールバック) → Stage 2 (CJKフォールバック) の順次フォールバックパイプラインを構築する。

### 3. WebAssembly環境への対応とスレッド処理の分離
- **問題**: `rayon` を用いたマルチスレッド処理やローカルファイルシステムへの直接アクセスは、標準的なWasmブラウザ環境ではサポートされない。
- **解決策**:
  - `cfg(target_arch = "wasm32")` を活用し、Wasmビルド時にはマルチスレッド抽出モジュールを非有効化。
  - シングルスレッド用のシーケンシャルな描画ループ・ドライバを導入し、`Trunk` を用いたビルド・デプロイ環境を整備する。

## 提案する変更内容

### `pdf_oxide_local/src/rendering/text_rasterizer.rs`
- 縦書き判定（`wmode == 1`）に基づく原点シフト `(v_shift_x, v_shift_y)` の適用。
- 縦書き用文字送り値の取得および座標更新ロジックの実装。
- 重複コードを排除したクリーンな順次フォールバックロジックへの統合。

### `src/main.rs` および `pdf_oxide_local/src/lib.rs`
- `target_arch = "wasm32"` 用の条件付きコンパイルによるスレッドプール・ファイルアクセスの抽象化。
- `HtmlCanvasElement` を活用したWebRunnerスタートアップの実装。
