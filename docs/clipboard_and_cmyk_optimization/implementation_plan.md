# 高解像度イメージ展開およびクリップボードコピーの高速化計画

サムネイルをクリックした後の「高解像度画像の生成」および「クリップボードへの転送データ作成」の処理を、ライブラリの活用と並列処理によって高速化します。

## 課題の現状
1.  **CMYK JPEG のデコード速度**: 高品質な PDF で多用される CMYK 形式の JPEG は、現在低速な `jpeg-decoder` クレートで展開されており、巨大な画像では数秒のレイテンシが発生します。
2.  **メインスレッドのブロッキング**: サムネイルクリック時にクリップボード用のデータ変換（RGBA → BGR、上下反転）がメインスレッドで行われており、高解像度画像（例: 300DPI）では UI が一時的にフリーズします。

## 提案する変更内容

### 1. `zune-jpeg` による CMYK → RGB 高速変換の導入
- **コンポーネント**: `pdf_oxide_local/src/extractors/images.rs`
- **内容**: すでに導入済みの `zune-jpeg` を拡張し、CMYK 形式の JPEG デコードにも適用します。
- **最適化**: `zune-jpeg` の `DecoderOptions` で出力色空間を `RGB` に指定することで、デコード処理と同時に SIMD 最適化された色空間変換を実行します。これにより、従来の `jpeg-decoder` + 手動変換パスに比べて数倍の高速化が見込まれます。

### 2. クリップボード用データ作成の高速化と非同期化 (Native/Windows)
- **コンポーネント**: `src/main.rs`
- **内容**: `copy_page` 関数内のピクセル変換処理を最適化します。
- **並列化**: **`rayon`** を活用し、ピクセルループ（RGBA → BGR 変換および DIB 形式への詰め替え）をマルチスレッドで並列実行します。
- **非同期化**: データの準備をバックグラウンドスレッドで実行するようにし、巨大な画像のコピー中も UI の操作性を維持します。

### 3. `bytemuck` 等によるメモリ操作の効率化
- **内容**: バッファのキャストやコピーに `bytemuck` を導入し、不要なメモリ確保や安全チェックのオーバーヘッドを削減します。

## ユーザーレビューが必要な事項 (User Review Required)

> [!NOTE]
> **色精度のトレードオフについて**
> `zune-jpeg` による CMYK → RGB 変換は非常に高速ですが、ICC プロファイルを用いない標準的な近似変換となります。厳密な色の再現性（印刷レベルの正確さ）よりも、表示とコピーの速度を優先する方針でよろしいでしょうか。

## 検証計画

### 自動テスト
- `cargo check` によるビルド確認。
- CMYK JPEG を含む PDF を用いたデコードテスト。

### 手動検証
- 300DPI 以上の高解像度でサムネイルをクリックし、UI のフリーズが解消されていること、およびコピー完了までの時間が短縮されていることを確認。
