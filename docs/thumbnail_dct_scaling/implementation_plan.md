# サムネイル作成時のJPEG高速デコード導入計画（更新版）

巨大なJPEG画像を含むPDFのサムネイル（プレビュー）作成時、デコード速度の向上および省メモリ化を図るため、より高速でビルド依存の少ないJPEGデコード処理を導入します。

## 当初計画からの変更理由と経緯
当初、C言語ライブラリに依存する `turbojpeg` クレートを用いたIDCTスケーリング（1/8縮小解凍）の導入を試みましたが、ユーザーのWindowsビルド環境においてC言語のコンパイルツールチェーン（`dlltool` 等）の欠如によるビルドエラーが発生しました。
ビルド環境の複雑化を防ぎ、かつ確実なパフォーマンス向上を実現するため、**純粋なRust実装として最速のJPEGデコーダである `zune-jpeg`** の導入へと方針を転換します。

## ユーザーレビューが必要な事項・確認の質問 (User Review Required)

> [!IMPORTANT]
> **方針転換の承認について**
> - `zune-jpeg` は純粋なRust実装でありながら、SIMD命令などをフル活用することで標準のデコーダよりも劇的に高速な展開が可能です。C言語ライブラリ依存がなくなり、WindowsやWASM環境へのビルド互換性が容易に維持できます。
> - 現時点の `zune-jpeg` のインターフェースでは明示的なIDCT間引き縮小オプションが制限されている場合がありますが、デコード速度自体の高速化により、巨大なスキャンPDF読み込み時のCPU負荷とレイテンシの改善目標を十分に達成可能です。この方針転換で進めてよろしいでしょうか。

## 提案する変更内容

### コンポーネント: `pdf_oxide` (ローカルPDFレンダリングエンジン)

#### [MODIFY] `pdf_oxide_local/Cargo.toml`
- 依存関係として純粋なRust実装の高速デコーダ `zune-jpeg = "0.4"`（または適切な最新バージョン）を追加します。
- WASMを含む全ターゲットで安全にビルド可能な構成とします。

#### [MODIFY] `pdf_oxide_local/src/extractors/images.rs`
- `PdfImage::to_dynamic_image` 内部のJPEG展開ロジックを改修します。
- 入力が `ImageData::Jpeg` の場合、標準の `image::load_from_memory` に代わって `zune_jpeg::JpegDecoder` を用いた超高速デコードパスを優先して実行するように変更します。
- 万が一デコードに失敗した場合の堅牢なフォールバックとして、既存のデコーダへの経路を維持します。

## 検証計画

### 自動テスト・ビルド検証
- `cargo check` および `cargo build` を実行し、Cビルドツールに依存せず正常にコンパイルが完了することを検証します。
- WASMターゲットへのクロスコンパイルがシームレスに通ることを確認します。

### 動作検証
- 巨大なJPEG画像を含むPDFのレンダリング処理を実行し、Pass 1（サムネイル生成）の処理速度向上を確認します。
- 展開された画像の色味や向きが正しく維持されていることを確認します。
